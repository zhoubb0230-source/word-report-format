## 1. 封面题目下要素：方正黑体_GBK/小三/固定行距29.4磅/两端对齐+首行缩进2字符；项目编号属此类

- **现象/坑**：① `checks.py` 早期注释写着 field 角色"leading spaces are template layout"，看到的人可能
  以为"保留行首空格"是设计意图；② 版式经历过多次变化——最初想"不动"、后改"删行首空格并**左对齐**+清首行
  缩进"、再到"**两端对齐 + 首行缩进2字符**"，字体也从"方正黑体"改成"**方正黑体_GBK**"、并新增"**行距固定值
  29.4磅**"。看到旧注释/旧提交会误以为该左对齐、该清掉首行缩进、字体是"方正黑体"或行距该跟正文的 28 磅；
  ③ **「项目编号」含"编号"二字**，直觉会以为它跟题目上方的"密级·文本编号行（仿宋16磅 classification）"是一类
  ——其实它是**题目下要素**，按 field 排版（方正黑体_GBK/小三）。
- **现在为什么这么做**：用户实测后要求：报告题目下的各要素（项目名称/项目编号/承担单位/项目负责人/起止时间/
  编制时间等）＝**方正黑体_GBK、小三（15磅=30半点）、行距固定值29.4磅（`lineRule=exact`,`line=588`）、两端
  对齐、首行缩进2字符**，同时**保留**"删除行首空格 + 清除左侧字符缩进"逻辑（行首空格/左缩进会把文本顶离左
  边距）。`_check_cover_field_left()` 设 `jc=both`、`set_first_line_chars=200`、清 `left/start` 系缩进（**不清
  首行缩进**，否则和 200 抵消）、删行首空格；`check_paragraph` field 分支另按 `spec.cover_field.line_twips`
  设固定行距。**行内填空空格（如「20   年   月」）必须保留**——那是填写位，不是版式。
  - **项目编号 → field（不是 classification）**：把泛化词「编号」**从 `spec.cover.classification_keywords`
    移除**，「项目编号：…」才不会被 `cover_role()` 关键词命中而误判成 classification（仿宋三号）。题目上方那条
    "文本编号/密级"行靠**位置** `above_title` 判 classification，与关键词无关，删「编号」不影响它。
  - **文本编号 ≠ 项目编号（曾错误当成同一项）**：`文本编号` 在**题目上方**、仿宋 **三号**（与密级一致，
    `spec.cover_classification`）；`项目编号` 在**题目下方**、方正黑体_GBK 小三（field）。两者**位置和字体字号
    都不同**，是**各自独立的必备项**（`required_fields` 同时含二者）。曾一度加过 `field_aliases`
    把「项目编号」并到「文本编号」的 presence 校验里——**已按用户更正删除**，`_cover_missing_fields` 回到各项
    独立子串匹配。别再把它们做成别名/互相顶替。
- **反例（别这么改）**：① 别改回"左对齐"或"清掉首行缩进"（更早一版，已被用户否掉）；② 别把 field 行**内**
  空格也 strip 掉（会毁填空位）——`_strip_para_ws(p, "leading")` 只清行首就是为此；③ 清缩进时只清 `left/start`、

- **关联**：`scripts/lib/checks.py` `_check_cover_field_left()`、field 分支行距、`_size_name`、

## 2. 封面题目必须删首尾空格后再居中；且左右缩进要一起清零

- **现象/坑**：题目已经 `jc=center` 了，为什么还要动文本 / 还要清右缩进？看着像多余操作。
- **现在为什么这么做**：① 题目首尾的空格会**参与居中计算**，导致视觉上偏移，不是真居中——所以
  `strip_text="both"` 删首尾空白（保留标题内部空格）后再居中。② `jc=center` 的居中基准是"左右缩进
  之后的可用宽度"，只清左缩进、留着右缩进（或反之）会让基准偏移，视觉上仍不居中——**左右必须一起清零**。
- **反例（别这么改）**：别只清单侧缩进；别为"省事"跳过 strip 直接居中。
- **关联**：`fbdc3d0`；`scripts/lib/checks.py`（title 分支 + `_check_no_indent`）。

## 3. "是否更新域"弹窗：最终选择保留 updateFields=true（接受一次弹窗）

- **现象/坑**：生成的文档打开时 Word 弹"是否更新域"，很像 bug。最直觉的"修法"是删掉
  `settings.xml` 里的 `<w:updateFields w:val="true"/>` 让弹窗消失——**这会重新引入真正的问题**。
- **现在为什么这么做**：目录(TOC)是域，正文被改后目录需要刷新才能对上。曾尝试"不设 updateFields
  也不打 dirty"来消弹窗(`bdaf1df`)，但那样目录不会自动刷新、会与改后的正文脱节；每字段的 `w:dirty`
  标记同样会触发**一模一样**的弹窗。最终权衡：**保留 `updateFields=true`**，Word 打开时弹一次、
  用户点「是」即重建目录与正文一致(`5505430`)。这是刻意接受的取舍，不是遗漏。
- **反例（别这么改）**：别为了消弹窗删 `updateFields`，也别改用 `w:dirty`——两者要么弹同样的窗、
  要么让目录失去自动刷新。
- **关联**：`10ba624`→`bdaf1df`→`5505430`；`scripts/40_apply_fixes.py` `_set_update_fields()`
  及其上方注释；只读排查工具 `scripts/diagnose_fields.py`。

## 4. 目录段落被排除在标题/图表判定与序号连续性之外

- **现象/坑**：目录里的条目文字长得就像各级标题，"为什么不识别它们"看着像漏判。**这问题反复复现过**：
  修了样式名（`2257429` 认「目录 N」本地化名），但"部分一级目录仍变黑体、缩进乱"又冒出来。
- **现在为什么这么做**：目录条目若参与标题识别，会被误判成标题（缩进错乱、字体被改成黑体等），还会污染
  序号连续性计数。所以 `cover/toc/body` 区域划分后，**TOC 区整体跳过**标题/图表判定与连续性。识别 TOC
  段落有三条信号，**缺一不可**：① 样式名/ID（`style_is_toc`，认 `TOC1`/「目录 N」）；② 段内 TOC 域指令
  （`has_toc_field`）；③ **TOC 域跨段范围**（`toc_field_para_indices`：`fldChar begin..end` 之间的所有段落）。
  ③ 是关键补丁——**Word 的目录是一个跨多段的域，只有第一条目带 `instrText "TOC"`，续行只有域"结果"**；
  若续行又继承了 `outlineLvl`（模板常见），只有 ①② 会把续行漏成大纲标题 → 黑体。③ 把整段域范围都标 TOC。
- **反例（别这么改）**：① 别"顺手"让目录段落走标题识别分支；② 别把 `toc_field_para_indices` 删成"只看
  当前段有没有 TOC 指令"——那正是续行漏判的根因；③ 处理嵌套域（条目内的 HYPERLINK/PAGEREF）要用深度栈，
  外层 TOC 要到自己的 `end` 才算闭合。
- **关联**：`2257429` → 本次 `toc_field_para_indices`；`scripts/20_extract_structure.py`
  （`toc_field_para_indices` + `is_toc` 三信号或运算）；`tests/test_e2e.py`
  `test_toc_field_span_tags_continuation_entries`。

## 5. 仅靠序号形状(pattern)匹配的标题/图表，绝不自动改文字，只批注

- **现象/坑**：明明检测到"3."「图1」这样的段落，为什么不直接按标题/图表规范改？看着像功能没做完。
- **现在为什么这么做**：纯形状正则误判风险高（普通正文也可能以「3.」「图1」开头）。只有 `outlineLvl` /
  标题·题注样式撑腰、或模型在第 2.5 步显式确认(`model_confirmed`)的，才 `直接改`;`level_source` /
  `source` 为 `pattern` 的**只生成批注**提示人工审查，绝不自动改其文字。
- **反例（别这么改）**：别把 pattern 级的标题/图表也纳入自动改文字/编号的分支——会误伤正文。
- **关联**：`SKILL.md` 严格边界第 9 条；`scripts/lib/checks.py`（`level_source != "pattern"` 判断）。

## 6. 过程件统一进唯一 workdir；清理是独立末步、成品必须落在 workdir 之外

- **现象/坑**：看到 `59_cleanup.py` 是单独一步，直觉会想"省事点，把清理合进 `50_finalize.py` 一步做完"；
  或看到 workdir 名字带随机后缀、`05_new_workdir.py` 像是多余的一层，想改成固定名 `workdir/` 图省事。
- **现在为什么这么做**：① workdir 用 `tempfile.mkdtemp` 建**唯一名**（`.word_report_work/run_<ts>_<rand>/`），
  就是为了**多 session 并发不互相覆盖**——固定名 `workdir/` 会让两个会话踩同一目录、内容串味（用户明确反馈过）。
  ② 清理**不能**合进 `50_finalize.py`：`50` 把成品复制到 `<out_dir>`，而**过程件、成品在物理上必须分离**——
  只有等成品与 `summary.md` 都到了 workdir **之外**、确认交付无误，才能删 workdir。合进 finalize 会造成"边输出
  边可能把还要用的目录删了"的时序风险，且 `55_summary.py` 在 `50` 之后还要读 workdir。所以顺序恒为
  `50 → 55 →（交付）→ 59`。③ `59` 带安全护栏（`lib/workdir.is_inside_base`），只删名字含 `.word_report_work`
  的目录、且不删基目录本身，避免误删。
- **反例（别这么改）**：① 别把 workdir 改成固定名或让模型自编路径（破坏并发隔离，退回"散乱/串味"）；
  ② 别把 `59` 的清理并进 `50`，也别把 `<out_dir>` 设在 workdir 里面（成品会被 `59` 一起删）；
  ③ 别为"删得干净"放宽 `is_inside_base` 的护栏去 rmtree 任意路径。
- **关联**：`scripts/05_new_workdir.py`、`scripts/59_cleanup.py`、`scripts/lib/workdir.py`；
  `SKILL.md` 第 0.5 步与两条 Path 的清理末步；`.gitignore` 的 `.word_report_work/`。

## 7. 西文字体只在【含西文的】正文/标题/表格强制；封面/目录不套；含西文时未设也改

- **现象/坑**：① 封面/目录西文没被改成 TNR，像"漏做"；② 曾经**每个纯中文段都误报**「西文字体应为
  Times New Roman（实际：仿宋）」——这是最初实现"未设也改、不看有没有西文"造成的（用户实测在表格上
  大量爆出，报"中文明明就是仿宋"）；③ 反过来，看到含西文段落 ascii 没设也被写 TNR，像"多此一举"。
- **现在为什么这么做**：`spec.western_font`（TNR）只对**正文/一~四级标题/表格内容**传 `western`；封面/目录
  **故意不传**（模板版式、点线号/页码）。**关键门控**：`_check_font_size(..., has_western=...)` 只在段落
  **确实含西文字符（拉丁字母或数字）**时才判西文——`rec.has_western` 由 `20` 从**全文**（非截断文本）算。
  纯中文段没有西文可规范，绝不因"生效 ascii=仿宋≠TNR"报错（中文 run 常把 ascii 也设成仿宋，不 gate 就
  段段误报）。含西文时才比对生效西文字体，不符或未设都写死 TNR（与"东亚字体未设也算不符"一致）。页码字体、
  目录超三级仍是**只提示不自动改**。
- **反例（别这么改）**：① 别把 `has_western` 门控删了退回"未设也一律改"（就是纯中文段海量误报的根因，
  用户已明确反馈）；② 别给封面/目录传 `western`；③ 别用**截断后的** `text[:60]` 算 has_western（西文可能
  在 60 字之后，会漏），要用全文；④ 别把页码/目录深度升级成自动改。
- **关联**：`scripts/lib/checks.py` `_check_font_size(western=, has_western=)`、正文/标题/表格调用点；
  `scripts/20_extract_structure.py` `rec.has_western`（`re.search(r"[A-Za-z0-9]", 全文)`）；
  `tests/test_checks.py` `TestWesternFont`（含纯中文表格不误判用例）；
  `spec/format_spec.json` `western_font`。

## 8. .doc 转换双后端：soffice 优先、Windows 回退 Microsoft Word

- **现象/坑**：① 早期只认 `soffice`，在**装了 Office 但没装 LibreOffice 的 Windows** 上，`.doc` 直接被判
  `can_convert_doc=false` 停掉，看着像"环境不支持"，其实本机 Word 完全能转；② 反过来，既然 Windows 有 Word，
  直觉会想"Windows 上优先用 Word"——别这么改；③ 探测 Word 时直觉会想"直接 `New-Object Word.Application`
  试一下能不能建"，这会**真的启动 Word 进程**、慢且可能弹 UI。
- **现在为什么这么做**：转换后端由 `scripts/lib/docconv.py` 统一封装，`can_convert_doc()` = soffice 或 msword
  任一可用；`convert()` **优先 soffice**（headless、无 Office UI、在沙箱/无桌面环境也能跑），**仅在 soffice
  缺席时**回退 `msword`。`msword.msword_available()` 用 PowerShell 查**注册表 ProgID**
  （`[Type]::GetTypeFromProgID('Word.Application')`）判断 Word 是否可自动化，**不启动 Word**；转换时才经
  PowerShell + COM 驱动 Word，且 `Documents.Open(src,$false,$true)` **只读打开**，绝不碰原件。走 PowerShell
  而非 pywin32 是为守"零第三方依赖"红线。
- **反例（别这么改）**：① 别在 Windows 上把优先级改成 Word 优先（soffice 更适合无头/沙箱，且行为跨平台一致）；
  ② 别把可用性探测改成"尝试实例化 COM 对象"（会启动 Word、慢/可能弹窗）；③ 别为省事引入 pywin32 依赖；
  ④ 别把 `.doc` 打开改成可写模式。
- **关联**：`scripts/lib/docconv.py`、`scripts/lib/msword.py`；`scripts/00_check_env.py`（多报 `msword`）、
  `scripts/10_prepare_input.py`、`scripts/50_finalize.py`（均改用 docconv 门面）；`tests/test_docconv.py`。

## 9. 分片 + 子 agent（Path B）已废弃：单一线性流程，别再加回来

- **现象/坑**：SKILL 曾把"支持子 agent 就必须走 Path B、把分片交给子 agent"列为**硬性要求**，理由写的是
  "避免 128k 窗口被 3000 页文档塞爆"。看着像大文档的必要优化，直觉会想"补回分片/子 agent 支持"。真实后果：
  有 agent（opencode）在第 3 步纠结"要不要用子 agent、凭什么选 Path A"，因为这个决策**根本没有可判据**——
  两条路本就产出同一份 `fixes.json`。
- **现在为什么这么做**：格式检查（`30_check_format.py`）是**纯 Python 全文扫描**——从磁盘读 `structure.json`、
  算出 `fixes.json`、只把计数摘要回给模型，**全程不把正文读进模型上下文**。所以：① 上下文窗口从不因文档大小
  受压，全量模式处理任意大小都没问题；② Path B 的子 agent **不做任何推理**，只是包一层跑同一个脚本再回传一个
