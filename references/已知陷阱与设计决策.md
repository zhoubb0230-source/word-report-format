# 已知陷阱与设计决策记录

> **用途**：只记一类知识——**"看起来像 bug、其实是刻意为之，别照直觉改回去"的回退陷阱**，
> 以及背后"我们试过 X、它坏了 Y、所以现在是 Z"的理由。
>
> **不记**：普通改动（看 git log）、某行的局部 why（看代码注释）、规范与工作流（看 `SKILL.md`）、
> 日常开发陷阱（看 `CLAUDE.md`）。
>
> **维护方式**：**只追加**。改代码时若发现自己正要撤销某条决策，先读这里；若新踩了"回退陷阱"级别的
> 坑，在文末加一条（现象 → 现在为什么这么做 → 反例：别怎么改 → 关联提交/文件）。**不要删旧条目**。

## 1. 封面字段行的行首空格：从"保留"改成"删除并左对齐"

- **现象/坑**：`checks.py` 早期注释明确写着 field 角色"leading spaces are template layout"，
  故意**不动**对齐与缩进。看到这句的人可能以为"保留行首空格"是设计意图，从而撤销现在的左对齐修复。
- **现在为什么这么做**：实测发现项目名称/承担单位/项目负责人/起止时间/编制时间这几行带行首空格时，
  视觉上没有左对齐，用户要求修复。现在 `_check_cover_field_left()` 会删行首空格、清左/首行缩进、
  必要时 `jc=left`。**行内填空空格（如「20   年   月」）必须保留**——那是填写位，不是版式。
- **反例（别这么改）**：① 别把 field 行首空格当"版式"保留回去；② 别顺手把行**内**空格也 strip 掉
  （会毁掉填空位）——`_strip_para_ws(p, "leading")` 只清行首就是为此。
- **关联**：`fbdc3d0`；`scripts/lib/checks.py` `_check_cover_field_left()`、
  `scripts/40_apply_fixes.py` `_strip_para_ws()`。

## 2. 封面题目必须删首尾空格后再居中；且左右缩进要一起清零

- **现象/坑**：题目已经 `jc=center` 了，为什么还要动文本 / 还要清右缩进？看着像多余操作。
- **现在为什么这么做**：① 题目首尾的空格会**参与居中计算**，导致视觉上偏移，不是真居中——所以
  `strip_text="both"` 删首尾空白（保留标题内部空格）后再居中。② `jc=center` 的居中基准是"左右缩进
  之后的可用宽度"，只清左缩进、留着右缩进（或反之）会让基准偏移，视觉上仍不居中——**左右必须一起清零**。
- **反例（别这么改）**：别只清单侧缩进；别为"省事"跳过 strip 直接居中。
- **关联**：`fbdc3d0`；`scripts/lib/checks.py`（title 分支 + `_check_no_indent`）。

## 3. "是否更新域"弹窗：最终选择保留 updateFields=true（接受一次弹窗）

- **现象/坑**：生成的文档打开时 Word 弹"是否更新域"，很像 bug。最直觉的"修法"是删掉
  `settings.xml` 里的 `<w:updateFields w:val="true"/>` 让弹窗消失——**这会重新引入真正的问题**。
- **现在为什么这么做**：目录(TOC)是域，正文被改后目录需要刷新才能对上。曾尝试"不设 updateFields
  也不打 dirty"来消弹窗(`bdaf1df`)，但那样目录不会自动刷新、会与改后的正文脱节；每字段的 `w:dirty`
  标记同样会触发**一模一样**的弹窗。最终权衡：**保留 `updateFields=true`**，Word 打开时弹一次、
  用户点「是」即重建目录与正文一致(`5505430`)。这是刻意接受的取舍，不是遗漏。
- **反例（别这么改）**：别为了消弹窗删 `updateFields`，也别改用 `w:dirty`——两者要么弹同样的窗、
  要么让目录失去自动刷新。
- **关联**：`10ba624`→`bdaf1df`→`5505430`；`scripts/40_apply_fixes.py` `_set_update_fields()`
  及其上方注释；只读排查工具 `scripts/diagnose_fields.py`。

## 4. 目录段落被排除在标题/图表判定与序号连续性之外

- **现象/坑**：目录里的条目文字长得就像各级标题，"为什么不识别它们"看着像漏判。
- **现在为什么这么做**：目录条目若参与标题识别，会被误判成标题（缩进错乱、字体被改成黑体等，
  见 `2257429`），还会污染序号连续性计数。所以 `cover/toc/body` 区域划分后，**TOC 区整体跳过**
  标题/图表判定与连续性。
- **反例（别这么改）**：别"顺手"让目录段落也走标题识别分支。
- **关联**：`2257429`；`SKILL.md`「识别封面/目录/正文区域」一节；`scripts/20_extract_structure.py`
  区域标记逻辑。

## 5. 仅靠序号形状(pattern)匹配的标题/图表，绝不自动改文字，只批注

- **现象/坑**：明明检测到"3."「图1」这样的段落，为什么不直接按标题/图表规范改？看着像功能没做完。
- **现在为什么这么做**：纯形状正则误判风险高（普通正文也可能以「3.」「图1」开头）。只有 `outlineLvl` /
  标题·题注样式撑腰、或模型在第 2.5 步显式确认(`model_confirmed`)的，才 `直接改`;`level_source` /
  `source` 为 `pattern` 的**只生成批注**提示人工审查，绝不自动改其文字。
- **反例（别这么改）**：别把 pattern 级的标题/图表也纳入自动改文字/编号的分支——会误伤正文。
- **关联**：`SKILL.md` 严格边界第 10 条；`scripts/lib/checks.py`（`level_source != "pattern"` 判断）。
