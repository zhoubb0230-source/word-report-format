# 格式审查与自动修正 (word-report-format)

本 skill 对一份中文自评价报告 Word 文档做**格式合规检查**并在**新文档**上自动修正，
给每处修改/提示加批注（批注内容即所违反的规范条目）。

## 严格边界（务必遵守）

1. **只做格式**：字体/字号/行距/缩进/页面设置/标题层级序号/图表序号连续性/表格内容字体字号。不做文字润色、不做逻辑审查。
2. **规范是固定的**：一切判定阈值只来自 `spec/format_spec.json`。**禁止**依据模型自身训练知识发挥，**禁止**引用其它 skill 的规范。
3. **绝不改原文**：所有改动都在解包出的副本上进行，原文件只读。
4. **先判断后修改**：脚本只对「当前生效格式 ≠ 规范」的项产出修改；符合的不动。
5. **不要让模型写脚本 / 手写 JSON**：所有格式判定、字体/字号/缩进/行距的取值、序号的生成规则全部由脚本产出，
   模型不得手工拼 `fixes.json`（避免引号未转义、字段名漏 `set_` 前缀等）。**唯一的例外**是第 2.5 步的
   `overrides.json`——schema 只有两个字段（段落序号 → 标题层级 / 图表种类），见下方说明；即便在那里，
   模型也只判断「这是几级标题/图还是表」，不产出任何格式数值或编号文本，那些仍由脚本按 spec 计算。
6. **输出命名**：`原文件名_格式化版本_时间戳.原扩展名`（由 `50_finalize.py` 负责，勿自行改名）。
7. **子 agent 委派是硬性要求**：若运行环境支持子 agent，必须走 Path B，把每个分片交给独立子 agent 处理（用于隔离上下文，避免 128k 窗口被 3000 页文档塞爆）。
8. **空行不处理**：空白/纯空格段落一律跳过格式判定与标题/图表识别——空行没有格式可言。
9. **标题层级与图表编号判定是全文级的、且允许模型判断，不能只靠正则**：正则/样式只能给出「形状上像什么」
   的默认猜测（例如「1.」形状默认猜三级标题），但同一份文档里「1.」到底是几级标题，只有看完全文的层级关系
   才能确定；而且标题可能完全没有可识别的编号或样式（正则/样式对这种情况天生看不见）。因此这一项判定
   **必须**先用 `26_export_review.py` 把全文的标题/图表候选、以及按字体信号找到的「可能漏标的标题」
   一次性导出（不能分片看，分片会看不到全局层级关系），模型通读后判断，需要修正/提升的写入 `overrides.json`
   交给 `27_apply_review.py` 应用；识别为已确认标题但漏了编号的，连续性检查会**按位置自动补号**（惯例
   不编号的章节如摘要/前言/结论/目录/参考文献/附录/致谢等除外，既不补号也不占序号位）。
   序号连续性、字体字号等仍然是确定性计算，不受此例外影响。
10. **「确认」和「未确认」的标题/图表标题，自动修改的尺度不同**：有 `outlineLvl` 或标题/题注样式撑腰的
    （`level_source`/`caption["source"]` 为 `outline`/`style`，或模型在第 2.5 步显式确认过、被标记为
    `model_confirmed`）——序号/层级有误时脚本**直接改**，不需要额外用户确认。**仅靠序号形状正则匹配、
    没有任何样式/大纲级别支撑**的（`level_source`/`source` 为 `pattern`）——脚本**绝不**改动这类段落的
    文字，只生成批注提示用户人工审查是否真的是标题/图表标题；只有模型在第 2.5 步明确把它判为真标题/
    图表标题并写入 `overrides.json`，它才会转为「已确认」，此后才可能被自动改。这是因为纯形状匹配的
    误判风险明显更高（普通正文偶尔也会以「3.」「图1」这样的文字开头）。
11. **图/表标题绝不会被误判为标题**：无论是否有 `outlineLvl`（哪怕模板遗留导致图表标题样式意外继承了
    大纲级别），只要文本形如「图/表+数字」或段落样式名包含题注类关键词（题注、Caption、图表标题等），
    一律先判定为图表标题候选，不会落入标题判定分支。

## 目录结构

```
word-report-format/
  SKILL.md                  ← 本文件
  spec/format_spec.json     ← 固定规范（唯一判定依据）
  scripts/
    00_check_env.py         ← 探测 python/lxml/soffice/msword，判断能否转换 .doc
    05_new_workdir.py       ← 创建本次运行专属的唯一工作目录（固定临时基目录下），所有过程件都落在这里
    10_prepare_input.py     ← 生成 working.docx（.doc 需转换）+ meta.json

    26_export_review.py     ← 导出全文标题/图表候选清单，供模型判断层级 → review_candidates.json
    27_apply_review.py      ← 把模型的层级/种类修正写回 structure.json（并重新生成 shards/）
    30_check_format.py      ← 格式检查（全量 或 --shard 单片）
    31_check_global.py      ← 全局项：页边距 + 序号连续性 + 内容提示
    35_merge_fixes.py       ← 合并各分片结果为 fixes.json
    40_apply_fixes.py       ← 在新文档上应用 + 加 XAgent 批注 → formatted.docx
    45_validate_output.py   ← 输出自检：zip完整/XML良构/必备part/段落数前后一致（坏了退2）
    50_finalize.py          ← 按命名规范输出（.doc 输入则转回 .doc）到 workdir 之外的输出目录
    55_summary.py           ← 从 fixes.json + apply_report.json 生成人类可读修改汇总表 summary.md
    59_cleanup.py           ← 交付完成后删除本次运行的工作目录（过程件全部清掉）
    diagnose_fields.py      ← 只读排查"是否更新域"弹窗来源（updateFields/dirty/外部引用域/外部关系/OLE）
    lib/                    ← 自包含依赖（lxml，无需 python-docx / defusedxml）
```

## 第 0 步：环境检查与分支（每次必做）

```bash
python scripts/00_check_env.py
```

输出单行 JSON：`{python, lxml, soffice, msword, can_convert_doc, ok}`。

- `ok=false` → 缺关键依赖（python 或 lxml），停止并告知用户环境不满足。
- `.doc` 转换支持两种后端，任一可用即 `can_convert_doc=true`：`soffice`（LibreOffice，跨平台）
  或 `msword`（Windows 上安装了 Microsoft Word/Office，经 COM 自动化调用）。
- 用户给的是 **.doc** 且 `can_convert_doc=false` → **停止**，通知用户：当前环境无法转换 .doc，
  无法保证输出类型一致，请改传 .docx，或在装有 LibreOffice（Linux/Mac）或 Microsoft Office
  （Windows）的环境运行。**不要**擅自改用别的格式硬跑。

## 第 0.5 步：创建本次运行的工作目录（每次必做）

```bash
python scripts/05_new_workdir.py
```

- 输出单行 JSON：`{status, workdir, base}`。**把 `workdir` 记下来，后续每一步的 `<workdir>` 参数都用它**，
  **不要自己另编一个路径**——所有过程件（`working.docx`/`structure.json`/`shards/`/`fixes.json`/
  `out_pkg/`/`formatted.docx`/`summary.md` 等）都统一落在这个目录里，不再散落各处。
- 该目录建在**固定临时基目录** `<当前工作目录>/.word_report_work/` 下、名字**每次运行都唯一**
  （`run_<时间戳>_<随机>`，由 `tempfile.mkdtemp` 原子创建）。因此**多个 session 并发也互不干扰**，
  各自有独立工作目录，不会串内容。（基目录可用环境变量 `WORD_REPORT_WORK_BASE` 改到别处。）
- `.word_report_work/` 已在 `.gitignore` 里，不会被提交。

## 第 1 步：准备输入

```bash
python scripts/10_prepare_input.py <输入文件> <workdir>
```

- .docx → 复制为 `<workdir>/working.docx`
- .doc → 用 LibreOffice 或（Windows 上）Microsoft Word 转成 `working.docx`；若不可转，退出码 2 并给出中文提示（见上）。
- 生成 `<workdir>/meta.json`（记录原名、原扩展名、working.docx 路径）。原文件始终只读。

## 第 2 步：抽取结构（解析「生效格式」）

```bash
python scripts/20_extract_structure.py <workdir> [--shard-size 400]
```

关键点（脚本已实现，模型无需关心细节）：
- 通过 `styles.xml` + `basedOn` 继承链解析**生效格式**（含继承来的 `outlineLvl`），不只看段落直接属性。
- 识别封面/目录/正文区域；**目录段落被排除**在标题/图表判定与连续性计数之外，避免假问题与污染。
- **封面报告标题**用全文级的"标题块"识别：标题常**换行成多段**（如「先进项目」/「2024年度自评价报告」），
  换行行可能丢了居中，逐段"看着像标题"会漏掉——脚本取封面上连续的大字号短行（排除带冒号的字段行、
  过长行），只要其中有一行是可信锚点（最大字号或居中），整块都打 `is_title`，从而每一行都会被居中并
  规范字体字号。
- **封面各要素按"角色"套格式**（封面无固定版式，可由第 2.5 步 AI 复核精确判定，见下）：
  - `title` 报告标题 → 方正小标宋 / 20磅 / 居中 / 清缩进 / **删除首尾多余空格**（首尾空格会参与居中导致
    视觉偏移，删后再居中；标题内部空格保留）；
  - `classification` 密级·文本编号行（同一行，**可能没有 key**，如「机密  202501323023」，**固定在题目
    上方**，靠位置 `above_title` 判定）→ 仿宋 / 16磅；
  - `field` 报告题目下的各要素（项目名称/**项目编号**/承担单位/项目负责人/起止时间/编制时间等）→
    **方正黑体_GBK / 小三（15磅）/ 行距固定值29.4磅** / **两端对齐、首行缩进2字符**（删除行首空格、
    清除左侧字符缩进；行内填空空格如「20   年   月」保留）。**注意「项目编号」属题目下要素、按 field 排版，
    不归 classification**——泛化词「编号」已从 `classification_keywords` 移除；presence 校验里「项目编号」
    经 `cover.field_aliases` 视作「文本编号」；
  - `other` → 不动。
  角色先由启发式默认（标题块 → title；**位于标题上方的行或含 `spec.cover.classification_keywords` 关键词**
  → classification；其余 → field），AI 复核可覆盖。字体字号值全部在 `spec`（`title`/`cover_classification`/
  `cover_field`），可调；密级行只校验字体字号、不动对齐缩进；字段行额外做两端对齐+首行缩进修复（见上）。
- **封面必备信息**（缺项提示、非自动改）：`项目名称`/`考核年份` 来自标题（常两行＝项目名称＋考核年份），
  密级可写成值（机密/秘密…）；确定性基线尽量不误报，AI 复核可用 `cover_present` 给出语义裁决权威覆盖；
  标题若仍是 `XXXX/模板` 占位符会单独提示未替换。
- 空白段落打 `is_blank`，不参与标题/图表识别与后续任何格式判定。
- 产出：`structure.json`（完整，**留在磁盘、不要读进模型上下文**）与 `shards/shard_NNN.json`（分片切片）。
- `structure.json` 里每条标题记录的 `level` 只是**形状/样式的默认猜测**，不是最终答案——下一步必做。
- 终端只打印计数摘要，供模型判断规模、决定走 Path A 还是 Path B。

## 第 2.5 步：标题/图表层级复核（模型判断，每次必做，不可分片）

```bash
python scripts/26_export_review.py <workdir>
```

- 产出 `<workdir>/review_candidates.json`，六份按文档顺序排列的清单，这个文件通常远小于全文，
  **直接读入模型上下文**审阅：
  - `headings_confirmed`：有 `outlineLvl` 或标题样式撑腰的标题候选，每条含当前的（形状猜测）层级、
    原文本、样式名/`outlineLvl`。这类**层级判对了就不用管**；判错了改一下会**直接自动应用**（无需
    进一步确认）。
  - `headings_unconfirmed`：**仅靠序号形状**（如「3.」「（一）」）匹配、既无 `outlineLvl` 也无标题样式
    的候选——纯文本形状匹配，脚本不确定这真的是标题（普通正文偶尔也会碰巧以「3.」开头）。这类脚本**绝不
    自动改**，只会在连续性检查时生成批注提示用户审查；只有在这里被模型明确判定为真标题并写进
    `overrides.json`，才会转为「已确认」，此后才可能被自动改。
  - `captions_confirmed` / `captions_unconfirmed`：图表标题同理——前者有题注类样式（题注/Caption/
    图表标题/图标题/表标题等）撑腰，自动改；后者仅靠「图/表+数字」文本形状匹配，只提示不改。`num_raw`
    为 `null` 表示样式已确认是图表标题、但整段没有任何编号（漏编号，而不是没识别出来）——这类在**平铺
    编号**时会被脚本按顺序**自动补**「图N/表N」（样式已确认，补号安全），仅在**章-序分组编号**时才退回
    批注提示（推不出属于哪一章）。`表标题` 样式的段落即使正文里没有「表」字（例如只写「设备清单」），也会
    被识别为表标题并补上「表N」。
  - `possible_missed_headings`：**不能只靠正则判断标题**——完全没被识别为标题（既无 `outlineLvl`、
    也无标题样式名、也没有任何可辨识的编号形状），但字体是规范里标题专属的字体（一级黑体/二级楷体，
    正文固定仿宋、不会用到这两种字体）的短行。这类多半是标题丢了编号或作者没编号，纯靠序号/样式正则
    永远发现不了，只能靠这个字体信号交给模型判断；同样只提示不改，除非被模型明确提升。
  - `cover_paragraphs`：**整页封面**的非空段落（封面无固定版式，是全页级判断），每条含原文本、字体、
    字号、对齐和启发式猜的角色 `guess_role`（`title`/`classification`/`field`）。逐条看猜得对不对：标题
    是否漏了/多认了行、密级·文本编号行有没有认出（该行**固定在标题上方**、**可能没 key**，只有
    「机密 202501…」这种——脚本已按"标题上方"位置默认判为 `classification`，但若版式特殊仍需你确认）。
    判错的写进 `overrides.json` 的 `cover` 段；`other` 可让某行**不被套用**任何封面格式。角色定了，
    字体字号仍由脚本按 `spec` 确定性套用。
  - `cover_required_fields` / `cover_missing_guess`：封面必备信息，以及**确定性基线**猜的缺项。基线对
    「项目名称/考核年份」这类**藏在标题里**、以及密级写成值（机密/秘密…）而非标签的情况会不准——**通读
    封面后按语义判断**：报告标题常分两行＝第一行项目名称、第二行考核年份（但要兼容其它写法），标题若还是
    `XXXX/模板` 占位符则视为未填。把**确实已填**的必备项写进 `overrides.json` 的 `cover_present`（数组），
    它会**权威覆盖**基线，决定"封皮缺少必要信息"提示。
- **必须通读全部候选后再下结论**——标题层级是全文级的结构判断：同一个「1.」在这份文档里到底是几级标题，
  只有看清楚它和前后其它标题的层级关系（有没有更高层的「一、」「（一）」把它们统领起来、序号是否在同一序列
  里连续递增）才能确定，脚本的形状猜测经常覆盖不了每份文档的实际写法。图表标题的分组（是否按章节前缀分组、
  一份文档里图和表各自的编号序列）同理，也要看完整份文档的图表列表再判断，不能只看单条。
- 逐条判断：`*_confirmed` 里这条猜测的层级/种类对不对（错了记下正确值，之后会自动改）？`*_unconfirmed`
  和 `possible_missed_headings` 里这条是不是真标题/真图表标题（是的话记下 `para_index` 和层级/种类去
  提升，不是就不用管——它本来就只是提示，不会被自动改）？
- 如果 `*_confirmed` 里的层级/种类都对、`*_unconfirmed`/`possible_missed_headings` 也没有需要提升的，
  **不需要**再做任何事，直接进入第 3 步（`31_check_global.py` 会照既有分类算连续性；已确认但漏了编号的
  标题/图表标题会在连续性检查时**按位置自动补号**（惯例不编号的章节除外，只跳过不补）；未确认的候选也会
  各自收到一条「疑似标题/图表标题，请人工确认」的提示）。
- 有需要修正/提升的，写一个 `overrides.json`（**只写要改的条目，其余保持默认**）：
  ```json
  {
    "headings": {"12": 1, "45": 2, "60": 1},
    "captions": {"30": "table"},
    "cover": {"3": "title", "1": "classification", "8": "field", "9": "other"},
    "cover_present": ["密级", "文本编号", "项目名称", "考核年份", "承担单位", "项目负责人", "起止时间", "编制时间"]
  }
  ```
  层级取值只能是 `1`/`2`/`3`/`4`/`null`（`null` = 这条其实不是标题）；图表种类只能是
  `"figure"`/`"table"`/`null`（`null` = 这条其实不是图表标题）；封面角色只能是
  `"title"`/`"classification"`/`"field"`/`"other"`/`null`（`null` = 清除覆盖、回退启发式）；
  `cover_present` 是**确实已填**的封面必备项数组（可选，给了就权威覆盖缺项基线）。`para_index`
  可以是文档里任意一个真实存在的段落序号——既能修正 `*_confirmed`/`*_unconfirmed`/`cover_paragraphs` 里
  已识别到的候选，**也能把 `possible_missed_headings` 里判断为真标题的段落直接提升为标题**（不局限于已被
  识别到的候选）。
  ```bash
  python scripts/27_apply_review.py <workdir> <overrides.json路径>
  ```
  该脚本会把修正写回 `structure.json`（**同时把这条标记为 `model_confirmed`，此后按「已确认」对待，
  可被自动改**）、**重新计算封面/正文边界**（提升/取消最早一条标题可能会改变这个边界——边界之前的段落
  此前会被当成「封面」而完全跳过标题/正文格式判定，边界一错，前面本该按正文处理的内容会被漏检）、并
  重建 `shards/`；下游的字体/字号/首行缩进/序号连续性检查会按**修正后**的层级重新计算——层级一变，
  适用的字体规范也跟着变（一级黑体、二级楷体、三/四级仿宋），所以修正层级之后格式判定是自动刷新的，
  不需要额外操作。

## 第 3 步：检查 + 应用（二选一）

### Path A —— 环境不支持子 agent（小文档亦可）

```bash
python scripts/30_check_format.py <workdir>      # 全量：逐段格式 + 页边距 + 连续性 + 内容提示 → fixes.json
python scripts/40_apply_fixes.py <workdir>       # 应用到新文档 + XAgent 批注 → formatted.docx
python scripts/45_validate_output.py <workdir>   # 输出自检；退出码非0（2）说明文档被改坏，勿交付、须排查
python scripts/50_finalize.py  <workdir> <输出目录>   # <输出目录> 必须在 workdir 之外（成品要留下来）
python scripts/55_summary.py   <workdir>         # 生成 summary.md 修改汇总表（连同 formatted 一起交付）
```

**交付完成后清理**（成品与 `summary.md` 已在 workdir 之外，过程件可以删了）：

```bash
python scripts/59_cleanup.py <workdir>           # 删除本次运行的工作目录，回收全部过程件
```

### Path B —— 支持子 agent（**优先/硬性要求**，适用于 3000 页大文档）

> 第 2.5 步的标题/图表层级复核**由主 agent 自己通读判断，不下放给分片子 agent**——它本来就是全文级判断，
> 分片会让子 agent 看不到跨分片的层级关系，判断不了。子 agent 只处理第 2 步里「逐段格式」这类不需要全文
> 上下文的机械检查。

1. 主 agent 先跑全局项（连续性必须跨全文，不能按片算）：
   ```bash
   python scripts/31_check_global.py <workdir>   # → fixes_parts/_global.json
   ```
2. **对 `shards/` 下每个分片，委派一个独立子 agent** 执行（互不共享上下文）：
   ```bash
   python scripts/30_check_format.py <workdir> --shard shards/shard_NNN.json
   # → fixes_parts/<片名>.json（子 agent 只需回传该片修改条数，不要回传全文）
   ```
3. 全部分片回来后，主 agent 合并并应用：
   ```bash
   python scripts/35_merge_fixes.py <workdir>    # 各片 + _global 合并去重 → fixes.json
   python scripts/40_apply_fixes.py <workdir>    # → formatted.docx（含 XAgent 批注）
   python scripts/45_validate_output.py <workdir># 输出自检；退出码非0（2）勿交付、须排查
   python scripts/50_finalize.py  <workdir> <输出目录>   # <输出目录> 必须在 workdir 之外
   python scripts/55_summary.py   <workdir>      # → summary.md 修改汇总表
   python scripts/59_cleanup.py   <workdir>      # 交付完成后删除工作目录，回收全部过程件
   ```

> 分片检查只做「逐段格式」；页边距与序号连续性是全局的，统一由 `31_check_global.py` 负责，
> 避免跨片时序号计数被切断。两条路径产出的 `fixes.json` 完全一致。

## 产物说明

- `summary.md`（`55_summary.py`）：人类可读的**修改汇总表**，与 `formatted.docx` 一并交付。它直接读
  `fixes.json` + `apply_report.json` 这两个**结构化**产物（不是反过来解析批注文字，所以批注措辞变了汇总也
  不会变空），按「位置 / 类别 / 内容摘要 / 规范依据 / 状态」列出每一处，并给出「已修正 / 提示待人工 /
  应用失败」计数。用户不必逐条翻文档批注就能一眼看全改了什么、哪些还需人工。
- `validate_report.json`（`45_validate_output.py`）：输出文档的结构自检结果。在 `40` 之后、`50` 之前跑，
  做 zip 完整性 + 必备 part 存在 + 每个 xml/rels 良构 + **段落数与改前 working.docx 一致** 四项检查
  （格式/编号/批注操作都不应增删段落，段落数变了即结构被改坏）。**退出码非 0（2）时不要交付**，先查
  `validate_report.json` 里的 `errors` 定位问题。这是防「真实 Word 打开提示需修复」的最后一道兜底。
- `formatted.docx`：已修正 + 每处改动/提示带 `XAgent` 批注。`40_apply_fixes.py` 设 `updateFields=true`，
  用户打开时 Word 会提示"该文档包含的域可能引用了其他文件。是否更新…"，**点"是"即自动刷新目录**（反映
  改后的标题序号与页码）。这个提示是 Word 处理带目录文档的**正常行为、非损坏**；`summary.md` 顶部已写明
  请点"是"。⚠️ Word 里"目录打开时自动刷新"与"零弹窗"**无法只靠文档设置两全**——全局 `updateFields` 和
  域级 `w:dirty` 都会触发同一个弹窗；唯一零弹窗的替代是不刷新、由用户 Ctrl+A→F9 手动刷。这是经确认的取舍，
  当前默认取"自动刷新（一次弹窗）"，**不要**擅自改回零弹窗方案。因目录刷新会按 TOC 样式重建条目并丢弃
  直接格式，`40_apply_fixes.py` 会同时把 `TOC1/TOC2/TOC3` 目录条目**样式本身**改成仿宋/14 磅/按级缩进
  （一级 0、二级 2 字符、三级 4 字符），保证刷新后目录仍符合规范。**若弹窗疑似另有来源**（原文档自带的
  INCLUDE·LINK·DDE 等外部引用域 / 外部关系 / OLE），用 `scripts/diagnose_fields.py <docx>` 排查定位。
- 图表标题排版为**居中、无任何缩进**；正文/标题修首行缩进时会把左、右缩进一并清零（避免与首行缩进叠加
  导致实际缩进超过 2 字符）；序号与标题正文之间用空格/制表符分隔（如「5 项目…」）也能识别并改正序号。
- **自动编号（Word 多级列表 `w:numPr`）**：
  - **标题**按「序号在不在文字里」区分——文字里没有序号（Word 生成的）就交给 Word 维护，不重排、不报
    漏编号（方案一）；文字里有**可选中的序号**（如手打「1    项目」）则视为手动序号，即使样式挂了自动
    编号也按规范改成「一、项目」。
  - **图/表标题**不再走方案一：已确认题注样式的图表标题，无论原来是手写编号、漏编号、还是挂了 Word
    自动编号，一律折进「图/表」各自的序列，改成**确定性的静态**「图N/表N」；对挂了自动编号的，应用阶段会
    在该段落写 `w:numPr → numId=0` 抵消样式级自动编号，再写死静态序号，从而把**异常/损坏的自动编号纠正**
    为正确的静态编号（而不是仅仅解除隐藏）。
  - 自动编号段落的缩进不一定完整写在 XML 里（二/三/四级常落到 Word 内置列表缩进），所以对自动编号
    标题/正文**强制**规范为「首行缩进2字符、左右0」，避免只有一级生效。
- 表格单元格内容有独立字体字号规范（仿宋 四号，`spec.table_body`）；脚本按 `in_table` 标记单独判定，
  且不会给表格内容强加首行缩进。
- **西文字体**：正文 / 一~四级标题 / 表格内容的**西文（拉丁字母、数字）**统一规范为 `spec.western_font`
  （Times New Roman）。**仅当该段落确实含西文字符时才判定**（`rec.has_western`，从全文而非截断文本算）——
  纯中文段落不判西文，避免每个中文段误报「西文应为 TNR」。含西文时比对生效西文字体，不符或未设都改。
  **封面与目录不强制**西文（封面属模板版式、目录西文多为页码/点线号）。
- `50_finalize.py` 输出：`<原名>_格式化版本_<时间戳>.<原扩展名>`；原为 .doc 时自动转回 .doc。
- 无法自动修的项（封面缺字段、封面绿色底、图表缺内容说明、疑似但未确认的标题/图表标题、
  章-序分组编号里漏编号的图表标题、**目录层级超三级**、**页码字体不符**）只以**批注提示**形式给出，不臆改。
  （目录深度取自目录条目样式末尾数字；页码字体在 `20` 阶段从页眉/页脚 `PAGE` 域尽力扫描，读到且不符才提示。）**已确认样式/大纲级别的标题**若漏了
  序号会**按位置自动补**（如三级补「1.」、一级补「一、」），惯例不编号的章节（摘要/前言/引言/结论/目录/
  参考文献/附录/致谢等）例外——既不补号也不占用同级序号计数。已确认题注样式、平铺编号但漏编号的图/表
  标题会**自动补**「图N/表N」序号（见上）。
- 仅靠序号形状识别、未经样式/大纲级别或模型确认的标题（`level_source="pattern"`），连字体/字号/缩进都
  **不会**按标题规范强制修改——脚本对这类段落按正文规范检查（就算它其实是正文，这样处理也不会错；万一
  它确实是标题，等模型在第 2.5 步确认后，下一轮就会按标题规范正确处理）。

## 模型在本流程中的职责（很重要）

模型主要做**编排**：按上面顺序调脚本、读计数摘要、决定 Path A/B、委派子 agent、把最终文件交付用户。
**唯一需要模型判断的地方是第 2.5 步**——通读 `review_candidates.json`，对 `*_confirmed` 里的候选判断
「层级/种类猜对了没有」，对 `*_unconfirmed`/`possible_missed_headings` 里的候选判断「这到底是不是真的
标题/图表标题」，因为这是全文级的结构关系，脚本的形状猜测覆盖不了所有文档的实际写法。除此之外：
**不要**：改写脚本、手写 `fixes.json`（或任何格式数值/编号文本）、凭训练知识判断字体字号等格式规范对错、
引入本 spec 之外的规则。字体/字号/缩进/行距/页边距的取值与序号连续性的计算，全部是脚本里的确定性逻辑，
不依赖模型推理——模型的判断只输入到「这是不是标题/图表标题、是第几级」这一个点，脚本据此重新计算出的
格式和编号仍然是确定性的；而且只有「确认」状态的标题/图表标题才会被脚本自动改动文字，未确认的永远只是
提示，不会因为脚本猜错而误改用户原文。
